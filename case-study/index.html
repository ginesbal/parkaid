1. The Challenge

Parking in downtown Calgary is surprisingly stressful. Information is scattered across street signs, city websites, and multiple third-party apps. Drivers end up circling blocks, unsure where spots are available, what the price is, or which zone rules apply.

The core question parkaid set out to answer:
"Where can I park right now, how much will it cost, and how far do I have to walk?"

The goal was to build a single, mobile-first tool that pulls together real parking data — locations, pricing, zone rules, and capacity — and presents it in a way that is fast to scan and easy to act on. No account required, no clutter, just the information a driver needs at the moment they need it.

2. Design Process

User Flow Design

The app is organized into three tabs that mirror the natural parking journey: first you browse options, then you locate one on a map, and finally you start your parking session.

BrowseHome — list view
→
LocateMap — spatial view
→
ParkSession — timer & cost

This three-step flow keeps the interface simple. Users don't need to learn the app — they just follow the tabs from left to right.

Progressive Disclosure

Instead of showing everything at once, parkaid reveals details in layers:

  • Home screen — shows the essentials: distance, price, and type for each spot.
  • Map screen — adds spatial context so you can see where spots are relative to your destination.
  • Flippable card — tap a marker and a card flips to reveal full details (zone info, capacity, hours, metadata).

This means the interface stays clean at first glance, but all the detail is there when you want it.

Key UI/UX Decisions

Warm Color Palette

A flame-red and earth-yellow scheme was chosen for high visibility and a sense of energy — appropriate for a "find parking now" tool.

Bottom Sheet on Map

A draggable bottom sheet shows the parking list over the map. This is a pattern users already know from Google Maps and Apple Maps, so there is no learning curve.

Pin-Drop Search

Long-press the map to drop a pin and search around that location. This lets users explore an area before they drive there.

Skeleton Loading

While data loads, animated skeleton placeholders keep the screen feeling responsive instead of showing a blank page or a spinner.

Information Architecture

The header on the Home screen includes quick stats — closest spot, cheapest option, total available count — so users get an instant snapshot without scrolling. Filters for spot type and search radius stay visible at the top, and a badge shows how many filters are active so nothing is hidden.

3. Research & Data

Calgary Open Data

parkaid uses real parking data from the City of Calgary's open data portal. An ETL (Extract, Transform, Load) pipeline pulls in spot locations, zone boundaries, capacity numbers, and pricing rules, then cleans and loads them into the database.

PostGIS for Spatial Queries

To answer "what's near me?", the app uses PostGIS — a geographic extension for PostgreSQL. It lets the backend run efficient radius-based searches (e.g., "find all spots within 500 meters of this latitude/longitude") without scanning every record in the table.

Google Maps & Places APIs

Google Maps renders the interactive map and markers. The Google Places API powers the search bar's autocomplete — type a street name or landmark and get instant suggestions. The API keys are protected on the backend (the mobile app never exposes them directly).

Design Assumptions

The app was designed for the "I need parking now" moment. That means speed and clarity come first: large tap targets, minimal navigation, and the most important info (distance, price) always visible. Feature density was deliberately kept low — the app does one thing well rather than many things poorly.

4. Testing & Validation

API Contract Tests

Backend tests verify that every endpoint returns the correct response structure and status codes, so frontend and backend stay in sync.

Performance Benchmarks

Spatial queries average around 120 ms. Slow queries (over 1 second) trigger warnings in the server logs so they can be investigated.

Data Quality Checks

The ETL pipeline validates parking data before it enters the database — bad coordinates, missing fields, or duplicate records are caught early.

Offline Resilience

If the network drops, the app falls back to cached data (up to 24 hours old). Users see slightly stale info rather than a blank screen.

Performance Tuning

  • Debounced API calls — a 300 ms delay before fetching prevents a flood of requests while the user is still adjusting filters.
  • Request deduplication — if the same request is already in flight, the app reuses the pending response instead of sending a second one.
  • Spatial indexing — PostGIS indexes on the geometry column keep radius queries fast even as the dataset grows.

State Management — Custom Hooks

Instead of using a heavy state library like Redux, parkaid manages state through four custom React hooks. Each hook owns one slice of the app's data and logic:

  • useLocationManager — handles GPS permissions, fetches the user's coordinates, reverse-geocodes to a street name, and caches the result locally.
  • useParkingSpots — calls the backend API to fetch nearby spots, debounces the request, and returns the list plus loading/error states.
  • useSessionManager — manages the parking session lifecycle (start, extend, end), runs a one-second timer for the countdown, calculates cost, and persists everything to local storage.
  • useFilterState — tracks the active spot-type filter and search radius, with an animated expand/collapse for the filter panel.

This approach keeps logic close to where it's used, avoids a central store that everything depends on, and makes each piece easy to test or replace independently.

API Design

The backend exposes a small set of RESTful endpoints:

  • GET /api/parking/nearby — accepts latitude, longitude, radius, and type; returns up to 100 spots sorted by distance with walking time and pricing info.
  • GET /api/parking/spot/:id — returns full details for a single spot.
  • GET /api/places/autocomplete — proxies the Google Places API so the mobile app never exposes the API key.
  • GET /api/places/details — fetches address and coordinates for a selected autocomplete result.

Notable Technical Patterns

  • API service class with built-in resilience — the mobile API layer includes a 5-minute cache, request deduplication, an 8-second timeout, and a fallback to stale cache on network failure. This means the app degrades gracefully instead of crashing.
  • Session state machine — a parking session moves through three states: active (normal countdown), expiring (under 10 minutes left — pulse animation kicks in), and expired. The session, including the timer, cost, and vehicle plate, is saved to local storage so it survives app restarts.
  • Flippable parking cards — when you tap a map marker, a card appears with a 3D flip animation. The front shows basic info; the back shows full details across scrollable pages. The card dynamically positions itself on screen based on the marker's coordinates.
  • Draggable bottom sheet — the map screen's parking list sits in a gesture-driven bottom sheet. As the sheet expands or collapses, the map adjusts its padding so markers aren't hidden behind it. Controls auto-hide after three seconds to keep the map uncluttered.

6. Collaboration with Developers

The codebase was structured so that multiple people could work on different parts without stepping on each other:

Modular Architecture

Each screen (Home, Map, Session) lives in its own folder with its own components, styles, and hooks. A developer working on the Map screen doesn't need to touch anything in the Session screen's folder. This reduces merge conflicts and makes code reviews faster.

Shared Design System

A centralized theme file defines color tokens, spacing values, and an alpha-transparency utility. Every component pulls from the same palette, so visual consistency is guaranteed without manual coordination. If the primary color changes, it changes everywhere.

Clear Separation of Concerns

  • Hooks handle business logic (fetching data, managing state, running timers).
  • Components handle the UI (rendering lists, cards, buttons, animations).
  • Services handle external communication (API calls, caching, error fallbacks).

This separation means a designer adjusting the UI doesn't need to understand the caching layer, and a backend developer can update the API service without touching any screen component.

Developer Onboarding

The project includes a README with setup instructions, a .env.example file so new developers know which environment variables are needed, and consistent naming conventions (folder structure, file naming, hook naming) that make the codebase predictable to navigate.

7. Key Challenges

  1. Spatial query performance
    Finding "all parking spots within 500 meters" sounds simple, but doing it efficiently with thousands of records requires spatial indexing. PostGIS's ST_DWithin function with a geometry index keeps queries under 200 ms. Without the index, the same query would scan every row in the table.
  2. Offline-first reliability
    Mobile networks are unreliable — tunnels, elevators, dead zones. parkaid uses a multi-layer cache: in-memory for instant re-reads, AsyncStorage for persistence across app restarts, and a stale-data fallback (up to 24 hours) when the network is completely unavailable. The user sees slightly outdated info instead of an error screen.
  3. Complex map interactions
    The map screen combines a draggable bottom sheet, a pin-drop search mode, animated markers, and flippable detail cards. Each of these involves its own set of Animated values and gesture handlers, and they all need to coordinate — for example, when the bottom sheet expands, the map padding adjusts so markers stay visible. Getting these animations to feel smooth at 60 fps required careful use of React Native's Animated API.
  4. Session state machine
    A parking session isn't just a timer. It moves through states (active, expiring, expired), calculates running cost, supports extensions (+15 min, +30 min, etc.), and needs to survive app restarts. The state machine runs on a one-second interval, updates the UI in real time, and persists every change to local storage so nothing is lost if the app is closed.
  5. Google Places API cost optimization
    Google charges per autocomplete request, but groups requests that share a session token into a single billing event. parkaid generates a session token and rotates it every three minutes, ensuring that a typical search-and-select flow is billed as one session instead of multiple individual requests.
